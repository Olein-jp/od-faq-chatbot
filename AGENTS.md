# AGENTS.md

このリポジトリは WordPress プラグイン開発用です。
Codex は以下の方針を常に守って作業してください。

## 基本方針

- WordPress Coding Standards に沿って実装する。
- 既存の設計、命名規則、ディレクトリ構成を優先する。
- 変更前に関連ファイルを確認し、影響範囲を把握してから作業する。
- 不要な大規模リファクタリングは行わない。
- 仕様が曖昧な場合は、既存実装に合わせて最小限かつ安全に実装する。

## 命名規則

- 独自関数、クラス、フック名、定数、オプション名、nonce 名、アクション名、フィルター名には、必ずこのプラグイン共通の接頭辞を使う。
- 接頭辞は既存コードから判断する。
- 既存の接頭辞が見つからない場合は、プラグインのスラッグをもとに一貫した接頭辞を提案してから実装する。
- WordPress コア、他プラグイン、テーマと衝突しやすい汎用的な名前は使わない。

## セキュリティ

- 入力値は必ず sanitize する。
- 出力値は必ず escape する。
- 権限チェックには `current_user_can()` を使う。
- フォーム送信、AJAX、REST API では nonce を検証する。
- SQL を直接書く場合は `$wpdb->prepare()` を使う。
- 外部リクエストでは WordPress HTTP API を使う。
- ファイル操作ではパス traversal や権限不足に注意する。

## 国際化

- 表示文字列は `__()`, `_e()`, `esc_html__()`, `esc_attr__()` などで翻訳可能にする。
- テキストドメインは既存実装に合わせる。
- 多言語展開を想定するプロダクトでは、コメント・ドキュメントは英語で書く。
- 日本語利用を主目的とするプロダクトでは、コメント・ドキュメントは日本語で書いてよい。
- ユーザー向け文言と開発者向けコメントを混同しない。

## コメント・ドキュメント

- PHPDoc、JSDoc、インラインコメントを適切に残す。
- なぜその実装にしたのか、将来の保守で迷いそうな箇所には理由を書く。
- 自明な処理に過剰なコメントは付けない。
- 公開関数、クラス、メソッド、フック、フィルターには説明を付ける。
- README、CHANGELOG、設定例、開発者向けメモが必要な変更では、関連ドキュメントも更新する。

## 品質チェック

変更後は、可能な限り以下を実行して問題がないことを確認する。

- PHP lint
- PHP_CodeSniffer
- WordPress Coding Standards
- JavaScript lint
- CSS lint
- HTML lint
- PHPUnit または既存のテスト
- npm scripts
- composer scripts

代表的なコマンド例:

```bash
find . -name "*.php" -not -path "./vendor/*" -not -path "./node_modules/*" -print0 | xargs -0 -n1 php -l
composer install
composer run lint
composer run phpcs
composer run test
npm install
npm run lint
npm run test
npm run build
```

既存の composer.json や package.json にスクリプトがある場合は、それを優先する。
存在しない場合は、プロジェクトに合う最小構成を提案・追加する。

## PHPCS / WPCS

* PHPCS と WordPress Coding Standards が未整備の場合は、導入できる構成を用意する。
* phpcs.xml.dist または phpcs.xml を追加・更新する。
* 対象外にすべきディレクトリは明示する。
    * vendor/
    * node_modules/
    * build/
    * dist/
* 自動修正できるものは phpcbf の利用も検討する。
* ただし、自動修正によって意味が変わる可能性がある場合は慎重に対応する。

## HTML / CSS / JavaScript

* HTML は適切なセマンティクスとアクセシビリティを意識する。
* CSS は既存の設計に合わせ、不要な詳細度の上昇を避ける。
* JavaScript は WordPress の標準的な書き方に合わせる。
* ブロックエディター関連では @wordpress/* パッケージの利用を優先する。
* グローバル変数の追加は避け、必要な場合は接頭辞を付ける。

## 依存パッケージ

* npm や composer のライブラリ・パッケージを追加・更新する場合は、原則として安定版の最新を利用する。
* 既存の依存関係との互換性を確認する。
* 不要な依存は追加しない。
* 小さな処理で済むものは、外部ライブラリを増やさず実装する。
* 依存追加時は、理由をコメントまたは作業報告に残す。

## WordPress 管理画面

* 管理画面 UI は WordPress 標準の見た目・操作感に寄せる。
* 権限チェック、nonce、sanitize、escape を必ず行う。
* 設定保存には Settings API の利用を優先する。
* 管理画面だけで使う CSS / JS は、必要な画面でのみ読み込む。

## REST API / AJAX

* REST API を追加する場合は permission callback を必ず定義する。
* AJAX ではログインユーザー向けと非ログインユーザー向けを明確に分ける。
* レスポンスは wp_send_json_success() / wp_send_json_error() などを使う。
* エラー時のメッセージは、利用者に伝わる内容にする。

## アセット管理

* CSS / JS は wp_enqueue_style() / wp_enqueue_script() で読み込む。
* バージョンにはプラグインバージョン、または開発時は filemtime() を使う。
* 不要な全ページ読み込みを避ける。
* ブロック用アセット、管理画面用アセット、フロント用アセットを分ける。

## データ保存

* オプション、メタデータ、独自テーブル、ファイル保存の使い分けを検討する。
* 保存前に sanitize する。
* 表示時に escape する。
* アンインストール時に削除すべきデータがある場合は uninstall.php を用意する。
* 破壊的なデータ削除は、ユーザーの明示的な設定または確認を前提にする。

## 後方互換性

* 対応 WordPress バージョン、PHP バージョンを意識する。
* 新しい PHP 構文を使う場合は、Requires PHP と矛盾しないか確認する。
* 既存のフック、フィルター、設定値、保存データの互換性を壊さない。

## 作業報告

作業完了時は以下を簡潔に報告する。

* 変更した主な内容
* 実行したチェック
* 成功したチェック
* 失敗したチェックと理由
* 未確認の項目
* 次に確認すべきこと

チェックを実行できなかった場合は、実行できなかった理由を明記する。

特に良いと思うのは、これを**全プラグイン共通のベース**にして、各リポジトリ側で「接頭辞」「テキストドメイン」「対応 PHP / WP バージョン」だけ追記する運用です。
